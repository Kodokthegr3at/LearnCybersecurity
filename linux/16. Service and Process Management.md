## 1. 🏢 Overview — What Are Services & Processes

### 🇮🇩 Bahasa Indonesia
**Service** (juga dikenal sebagai **daemon**) adalah program yang berjalan di **background tanpa interaksi langsung dari pengguna**. Service dibagi menjadi dua kategori:

- **System Services** — Service internal yang dibutuhkan saat sistem booting. Menangani tugas hardware penting dan inisialisasi komponen OS. Seperti mesin dan transmisi pada mobil — sistem tidak bisa berjalan tanpanya.
- **User-Installed Services** — Service yang ditambahkan pengguna, seperti aplikasi server atau proses background lainnya. Seperti AC atau GPS pada mobil — tidak wajib, tapi menambah fungsionalitas.

Daemon biasanya diidentifikasi dengan huruf **`d`** di akhir nama program: `sshd`, `systemd`, `httpd`, dll.

**Tujuan umum saat berinteraksi dengan service/process:**
- Start / Restart service atau process
- Stop service atau process
- Melihat status dan log service atau process
- Enable / Disable service saat boot
- Mencari service atau process yang sedang berjalan

### 🇬🇧 English
**Services** (also known as **daemons**) are programs that run **in the background without direct user interaction**. Services fall into two categories:

- **System Services** — Internal services required during system startup. They handle essential hardware tasks and OS initialization. Like the engine and transmission of a car — the system can't run without them.
- **User-Installed Services** — Services added by users, such as server applications or other background processes. Like a car's AC or GPS — not critical, but enhance functionality.

Daemons are typically identified by the letter **`d`** at the end of their program name: `sshd`, `systemd`, `httpd`, etc.

**Common goals when dealing with services/processes:**
- Start / Restart a service or process
- Stop a service or process
- View the status and logs of a service or process
- Enable / Disable a service at boot
- Find a running service or process

### 🇯🇵 日本語
**サービス**（**デーモン**とも呼ばれる）は**ユーザーとの直接的なやり取りなしにバックグラウンドで動作する**プログラムです。サービスは2つのカテゴリに分かれます：

- **システムサービス** — システム起動時に必要な内部サービス。ハードウェアの重要なタスクとOSの初期化を担当します。車のエンジンやトランスミッションのようなもの — これなしではシステムは動きません。
- **ユーザーインストールサービス** — ユーザーが追加するサービス（サーバーアプリや他のバックグラウンドプロセスなど）。車のエアコンやGPSのようなもの — 必須ではないが機能を向上させます。

デーモンは通常、プログラム名の末尾に **`d`** が付いています：`sshd`、`systemd`、`httpd` など。

**サービス/プロセスを扱う際の一般的な目的：**
- サービスやプロセスの起動・再起動
- サービスやプロセスの停止
- サービスやプロセスのステータスとログの確認
- 起動時のサービスの有効化・無効化
- 実行中のサービスやプロセスの検索

| State | 🇮🇩 Deskripsi | 🇬🇧 Description | 🇯🇵 説明 |

## 2. 🔧 `systemctl` — Control Services with systemd

### 🇮🇩 Bahasa Indonesia
`systemctl` adalah perintah utama untuk **mengontrol systemd**, init system yang digunakan oleh sebagian besar distro Linux modern. systemd adalah **proses pertama yang berjalan saat booting** dan diberi PID 1. Semua proses di Linux memiliki PID dan dapat dilihat di direktori `/proc/`.

### 🇬🇧 English
`systemctl` is the primary command for **controlling systemd**, the init system used by most modern Linux distributions. systemd is the **first process that starts during boot** and is assigned PID 1. All processes in Linux are assigned a PID and can be viewed under the `/proc/` directory.

### 🇯🇵 日本語
`systemctl` は**systemdを制御する**主要なコマンドです。systemdはほとんどの現代的なLinuxディストリビューションで使われるinitシステムです。systemdは**起動時に最初に動作するプロセス**でありPID 1が割り当てられます。LinuxのすべてのプロセスにはプロセスIDが割り当てられ、`/proc/` ディレクトリで確認できます。

```bash
# Start a service
$ systemctl start ssh

# Stop a service
$ systemctl stop ssh

# Restart a service
$ systemctl restart ssh

# Check the status of a service
$ systemctl status ssh

● ssh.service - OpenBSD Secure Shell server
   Loaded: loaded (/lib/systemd/system/ssh.service; enabled)
   Active: active (running) since Thu 2020-05-14 15:08:23 CEST; 24h ago
   Main PID: 846 (sshd)
   ...

# Enable a service to start at boot
$ systemctl enable ssh

# Disable a service from starting at boot
$ systemctl disable ssh

# List all running services
$ systemctl list-units --type=service

UNIT                     LOAD   ACTIVE SUB     DESCRIPTION
accounts-daemon.service  loaded active running Accounts Service
apache2.service          loaded active running The Apache HTTP Server
avahi-daemon.service     loaded active running Avahi mDNS/DNS-SD Stack
...

# Check if a service is running (use with ps)
$ ps -aux | grep ssh

root  846  0.0  0.1  72300  5660 ?  Ss  Mai14  0:00 /usr/sbin/sshd -D
```

| Command | 🇮🇩 Fungsi | 🇬🇧 Function | 🇯🇵 機能 |
|---------|-----------|-------------|---------|
| `systemctl start <svc>` | Jalankan service | Start a service | サービスを起動 |
| `systemctl stop <svc>` | Hentikan service | Stop a service | サービスを停止 |
| `systemctl restart <svc>` | Restart service | Restart a service | サービスを再起動 |
| `systemctl status <svc>` | Lihat status service | View service status | サービスのステータスを確認 |
| `systemctl enable <svc>` | Aktifkan saat boot | Enable at boot | 起動時に有効化 |
| `systemctl disable <svc>` | Nonaktifkan saat boot | Disable at boot | 起動時に無効化 |
| `systemctl list-units` | Daftar semua service | List all services | 全サービスを一覧表示 |

> 💡 **Pentesting note:** `systemctl enable` adds the service to SysV startup scripts, ensuring it persists across reboots — useful when setting up persistent listeners or pivoting infrastructure.

## 3. 📋 `journalctl` — View Service Logs

### 🇮🇩 Bahasa Indonesia
`journalctl` digunakan untuk **melihat log dari systemd journal**. Sangat berguna ketika service gagal start — kamu bisa melihat error yang terjadi secara detail, termasuk log dari sesi sebelum reboot.

### 🇬🇧 English
`journalctl` is used to **view logs from the systemd journal**. Especially useful when a service fails to start — you can see detailed error messages, including logs from sessions before a reboot.

### 🇯🇵 日本語
`journalctl` は**systemdジャーナルのログを表示する**コマンドです。サービスの起動に失敗したときに特に便利で、再起動前のセッションのログを含む詳細なエラーメッセージを確認できます。

```bash
# View logs for a specific service
$ journalctl -u ssh.service --no-pager

-- Logs begin at Wed 2020-05-13 17:30:52 CEST --
Mai 13 20:38:44 inlane systemd[1]: Starting OpenBSD Secure Shell server...
Mai 13 20:38:44 inlane sshd[2722]: Server listening on 0.0.0.0 port 22.
Mai 13 20:39:27 inlane sshd[3942]: Accepted password for master from 10.22.2.1 port 36452
Mai 14 02:04:49 inlane systemd[1]: Stopped OpenBSD Secure Shell server.
-- Reboot --

# Follow logs in real-time (like tail -f)
$ journalctl -u ssh.service -f

# View logs since last boot only
$ journalctl -u ssh.service -b

# View logs with timestamps
$ journalctl -u ssh.service --since "2020-05-14" --until "2020-05-15"

# View kernel logs
$ journalctl -k

# View all logs (no pager)
$ journalctl --no-pager
```

> ⚠️ **Tip:** If a service fails to start, always check `journalctl -u <service>.service` before anything else — it tells you exactly why it failed.

## 4. ☠️ Kill a Process — Sending Signals

### 🇮🇩 Bahasa Indonesia
Proses dapat dikontrol dengan mengirimkan **sinyal** menggunakan perintah `kill`, `pkill`, `pgrep`, dan `killall`. Setiap sinyal memiliki nomor dan fungsi yang berbeda. Sinyal yang paling sering digunakan dalam pentesting adalah **SIGKILL (9)** untuk memaksa kill dan **SIGTERM (15)** untuk terminasi normal.

### 🇬🇧 English
Processes are controlled by sending **signals** using the `kill`, `pkill`, `pgrep`, and `killall` commands. Each signal has a number and a specific function. The most commonly used signals in pentesting are **SIGKILL (9)** to force kill and **SIGTERM (15)** for graceful termination.

### 🇯🇵 日本語
プロセスは `kill`、`pkill`、`pgrep`、`killall` コマンドを使って**シグナル**を送ることで制御します。各シグナルには番号と特定の機能があります。ペンテストで最もよく使われるシグナルは強制終了の **SIGKILL (9)** と正常終了の **SIGTERM (15)** です。

```bash
# List all available signals
$ kill -l

 1) SIGHUP    2) SIGINT    3) SIGQUIT   9) SIGKILL  15) SIGTERM
19) SIGSTOP  20) SIGTSTP  ...

# Kill a process by PID (graceful - SIGTERM is default)
$ kill <PID>

# Force kill a process (SIGKILL - no cleanup)
$ kill -9 <PID>

# Kill by process name
$ pkill firefox

# Kill by name (force)
$ pkill -9 firefox

# Find PID by name
$ pgrep ssh
846

# Kill all instances of a process
$ killall firefox

# Find process details before killing
$ ps -aux | grep <processname>
```

## 5. 🔑 Common Signals Reference

| Signal | # | Keyboard | 🇮🇩 Deskripsi | 🇬🇧 Description | 🇯🇵 説明 |
|--------|---|----------|--------------|----------------|---------|
| `SIGHUP` | 1 | — | Terminal ditutup | Terminal controlling it closed | 制御端末が閉じられた |
| `SIGINT` | 2 | `Ctrl + C` | Interrupt dari pengguna | User interrupt | ユーザーによる割り込み |
| `SIGQUIT` | 3 | `Ctrl + D` | Quit dari pengguna | User quit | ユーザーによる終了 |
| `SIGKILL` | 9 | — | Paksa kill (tidak bisa ditangkap) | Force kill (cannot be caught) | 強制終了（捕捉不可） |
| `SIGTERM` | 15 | — | Terminasi normal | Normal termination | 正常終了 |
| `SIGSTOP` | 19 | — | Stop proses (tidak bisa ditangkap) | Stop process (cannot be caught) | プロセス停止（捕捉不可） |
| `SIGTSTP` | 20 | `Ctrl + Z` | Suspend proses (bisa dilanjutkan) | Suspend process (can be resumed) | プロセスを一時停止（再開可能） |

## 6. 🌑 Background a Process

### 🇮🇩 Bahasa Indonesia
Saat menjalankan scan atau proses yang memakan waktu lama, kita mungkin ingin **memindahkannya ke background** agar bisa tetap menggunakan terminal yang sama. Ada dua cara: menggunakan `Ctrl + Z` lalu `bg`, atau menambahkan `&` di akhir perintah.

### 🇬🇧 English
When running a long scan or process, you may want to **move it to the background** to keep using the same terminal session. There are two ways: use `Ctrl + Z` then `bg`, or add `&` at the end of the command.

### 🇯🇵 日本語
時間のかかるスキャンやプロセスを実行するとき、同じターミナルセッションを使い続けるために**バックグラウンドに移動**したい場合があります。方法は2つ：`Ctrl + Z` を押してから `bg` を使う方法と、コマンドの末尾に `&` を追加する方法です。

```bash
# Method 1: Suspend then background
$ ping -c 10 www.hackthebox.eu
PING www.hackthebox.eu (104.20.55.68) 56(84) bytes of data.
[Ctrl + Z]
[1]+  Stopped    ping -c 10 www.hackthebox.eu

# Send the stopped process to background
$ bg
[1]+ ping -c 10 www.hackthebox.eu &

# Method 2: Start directly in background with &
$ ping -c 10 www.hackthebox.eu &
[1] 10825
PING www.hackthebox.eu (172.67.1.1) 56(84) bytes of data.

# View all background jobs
$ jobs
[1]+  Running    ping -c 10 www.hackthebox.eu &
[2]+  Stopped    vim tmpfile
```

> 💡 **Pentesting note:** Use `nmap -sV target &` to run scans in the background while continuing enumeration in the same terminal.

## 7. 🌕 Foreground a Process

### 🇮🇩 Bahasa Indonesia
Untuk **mengembalikan proses dari background ke foreground** dan berinteraksi kembali dengannya, gunakan perintah `fg` diikuti nomor job. Nomor job bisa dilihat dari output perintah `jobs`.

### 🇬🇧 English
To **bring a background process back to the foreground** and interact with it again, use the `fg` command followed by the job number. Job numbers are visible in the output of the `jobs` command.

### 🇯🇵 日本語
バックグラウンドのプロセスを**フォアグラウンドに戻して**再び操作するには、`fg` コマンドにジョブ番号を続けて使います。ジョブ番号は `jobs` コマンドの出力で確認できます。

```bash
# List all background jobs
$ jobs
[1]+  Running    ping -c 10 www.hackthebox.eu &
[2]-  Stopped    vim tmpfile

# Bring job #1 to foreground
$ fg 1
ping -c 10 www.hackthebox.eu

--- www.hackthebox.eu ping statistics ---
10 packets transmitted, 0 received, 100% packet loss, time 9206ms

# Bring job #2 (vim) to foreground
$ fg 2
```

## 8. ⛓️ Execute Multiple Commands

### 🇮🇩 Bahasa Indonesia
Ada tiga cara untuk menjalankan beberapa perintah secara berurutan, masing-masing dengan perilaku berbeda tergantung pada hasil perintah sebelumnya:

- **Semicolon `;`** — Jalankan semua perintah berurutan, **abaikan error**
- **Double Ampersand `&&`** — Jalankan perintah berikutnya **hanya jika yang sebelumnya berhasil**
- **Pipe `|`** — Kirim **output** perintah pertama sebagai **input** perintah berikutnya

### 🇬🇧 English
There are three ways to run multiple commands sequentially, each with different behavior depending on the result of the previous command:

- **Semicolon `;`** — Runs all commands in sequence, **ignores errors**
- **Double Ampersand `&&`** — Runs the next command **only if the previous succeeded**
- **Pipe `|`** — Passes the **output** of the first command as **input** to the next

### 🇯🇵 日本語
複数のコマンドを順番に実行する方法は3つあり、それぞれ前のコマンドの結果によって動作が異なります：

- **セミコロン `;`** — すべてのコマンドを順番に実行し、**エラーを無視する**
- **ダブルアンパサンド `&&`** — 前のコマンドが**成功した場合のみ**次のコマンドを実行
- **パイプ `|`** — 最初のコマンドの**出力**を次のコマンドの**入力**として渡す

```bash
# ── SEMICOLON (;) — ignores errors ──────────────────────────
# Runs all 3 commands regardless of results
$ echo '1'; echo '2'; echo '3'
1
2
3

# Even with an error in the middle, the 3rd command still runs
$ echo '1'; ls MISSING_FILE; echo '3'
1
ls: cannot access 'MISSING_FILE': No such file or directory
3

# ── DOUBLE AMPERSAND (&&) — stops on error ──────────────────
# Stops at the failed command — echo '3' is never reached
$ echo '1' && ls MISSING_FILE && echo '3'
1
ls: cannot access 'MISSING_FILE': No such file or directory

# Only runs next command if previous succeeded
$ sudo apt update && sudo apt upgrade -y

# ── PIPE (|) — chains output to input ───────────────────────
# Sends output of ps to grep for filtering
$ ps -aux | grep ssh

root  846  0.0  0.1  72300  5660 ?  Ss  Mai14  0:00 /usr/sbin/sshd -D

# Chain multiple pipes
$ cat /etc/passwd | grep "/bin/bash" | cut -d: -f1
```

| Separator | 🇮🇩 Perilaku | 🇬🇧 Behavior | 🇯🇵 動作 |
|-----------|-------------|-------------|---------|
| `;` | Selalu jalankan semua, abaikan error | Always runs all, ignores errors | 常にすべて実行、エラー無視 |
| `&&` | Berhenti jika ada error | Stops at any error | エラーで停止 |
| `\|` | Output → Input berikutnya | Output → Next input | 出力→次の入力へ |

## 🔗 Practical Workflow Example

```bash
# ── SCENARIO: Set up and manage SSH service ──────────────────

# 1. Install OpenSSH server
$ sudo apt install openssh-server -y

# 2. Start the SSH service
$ sudo systemctl start ssh

# 3. Verify it's running
$ systemctl status ssh

# 4. Enable SSH to start on every boot
$ sudo systemctl enable ssh

# 5. Confirm it's running using ps
$ ps -aux | grep ssh

# 6. Check logs for any errors
$ journalctl -u ssh.service --no-pager

# 7. Run a long scan in background while SSH is up
$ nmap -sV -p- 10.10.10.1 &

# 8. Check background jobs
$ jobs

# 9. Bring the nmap scan back to foreground if needed
$ fg 1

# 10. If a process hangs, find its PID and force kill it
$ pgrep nmap
12345
$ kill -9 12345

# 11. Restart SSH after config changes
$ sudo systemctl restart ssh

# 12. Disable SSH when no longer needed
$ sudo systemctl disable ssh
$ sudo systemctl stop ssh
```

## 🧠 Quick Reference Cheatsheet

```bash
# ── SYSTEMCTL ────────────────────────────────────────────────
sudo systemctl start <service>           # start service
sudo systemctl stop <service>            # stop service
sudo systemctl restart <service>         # restart service
systemctl status <service>               # check status
sudo systemctl enable <service>          # enable on boot
sudo systemctl disable <service>         # disable on boot
systemctl list-units --type=service      # list all services

# ── JOURNALCTL ───────────────────────────────────────────────
journalctl -u <service>.service          # view service logs
journalctl -u <service>.service -f       # follow logs live
journalctl -u <service>.service -b       # logs since last boot
journalctl -u <service> --no-pager       # no pagination

# ── PROCESS SIGNALS ──────────────────────────────────────────
kill -l                                  # list all signals
kill <PID>                               # SIGTERM (graceful)
kill -9 <PID>                            # SIGKILL (force)
pkill <name>                             # kill by name
pkill -9 <name>                          # force kill by name
pgrep <name>                             # find PID by name
killall <name>                           # kill all instances
ps -aux                                  # list all processes
ps -aux | grep <name>                    # find specific process

# ── BACKGROUND / FOREGROUND ──────────────────────────────────
<command> &                              # start in background
[Ctrl + Z]                               # suspend current process
bg                                       # resume in background
fg <job_id>                              # bring to foreground
jobs                                     # list background jobs

# ── MULTIPLE COMMANDS ────────────────────────────────────────
cmd1 ; cmd2 ; cmd3                       # run all (ignore errors)
cmd1 && cmd2 && cmd3                     # run next only if success
cmd1 | cmd2 | cmd3                       # pipe output to next cmd

# ── COMMON COMBOS ────────────────────────────────────────────
sudo apt update && sudo apt upgrade -y   # update then upgrade
ps -aux | grep ssh                       # find SSH process
journalctl -u ssh -f                     # live SSH logs
nmap -sV target &                        # scan in background
```
