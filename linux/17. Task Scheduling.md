## 1. 🏢 Overview — Why Task Scheduling Matters

### 🇮🇩 Bahasa Indonesia
**Task scheduling** adalah fitur penting di Linux yang memungkinkan pengguna dan administrator untuk **mengotomatisasi tugas** dengan menjalankannya pada waktu tertentu atau interval yang ditentukan — tanpa perlu memulainya secara manual.

Contoh penggunaan umum:
- Pembaruan software otomatis
- Eksekusi script terjadwal
- Maintenance database berkala
- Otomatisasi backup

Analogi mudah: task scheduling seperti **menyetel timer kopi di pagi hari** — sekali diprogram, kopi tersedia tepat waktu tanpa intervensi lebih lanjut.

**Relevansi untuk Cybersecurity:**
Sebagai penetration tester, memahami task scheduling sangat penting karena:
- Bisa menjadi **vektor serangan** — cron job berbahaya dapat mengeksekusi backdoor secara berkala
- Membantu **deteksi persistensi** — attacker sering menyembunyikan persistence via cron
- Berguna untuk **simulasi serangan** saat pentest

### 🇬🇧 English
**Task scheduling** is a critical Linux feature that allows users and administrators to **automate tasks** by running them at specific times or defined intervals — without manual initiation.

Common use cases:
- Automatic software updates
- Scheduled script execution
- Periodic database maintenance
- Backup automation

Easy analogy: task scheduling is like **setting a coffee maker timer in the morning** — once programmed, coffee is ready on time without further intervention.

**Cybersecurity Relevance:**
As a penetration tester, understanding task scheduling is critical because:
- It can be an **attack vector** — malicious cron jobs can execute backdoors at regular intervals
- It aids in **persistence detection** — attackers often hide persistence via cron
- Useful for **attack simulation** during pentests

### 🇯🇵 日本語
**タスクスケジューリング**はLinuxの重要な機能で、ユーザーや管理者が特定の時刻や定義された間隔でタスクを実行することで**自動化**できます — 手動での操作なしに。

一般的な使用例：
- ソフトウェアの自動更新
- スクリプトのスケジュール実行
- 定期的なデータベースメンテナンス
- バックアップの自動化

わかりやすい例え：タスクスケジューリングは**朝のコーヒーメーカータイマーをセットする**ようなもの — 一度プログラムすれば、それ以上の操作なしに時間通りにコーヒーが用意されます。

**サイバーセキュリティとの関連：**
ペンテスターとして、タスクスケジューリングを理解することは重要です：
- **攻撃ベクター**になり得る — 悪意のあるcronジョブが定期的にバックドアを実行する可能性がある
- **永続性の検出**に役立つ — 攻撃者はcronを通じて永続性を隠すことが多い
- ペンテスト中の**攻撃シミュレーション**に活用できる

## 2. 🔧 Systemd Timer — Schedule Tasks with systemd

### 🇮🇩 Bahasa Indonesia
**Systemd** dapat digunakan untuk menjalankan proses atau script pada waktu atau interval tertentu. Berbeda dengan cron, systemd menggunakan **dua file terpisah**: sebuah **timer** (kapan dijalankan) dan sebuah **service** (apa yang dijalankan).

Langkah-langkah membuat task terjadwal dengan systemd:
1. Buat file **timer** (`.timer`) — mendefinisikan kapan service dijalankan
2. Buat file **service** (`.service`) — mendefinisikan apa yang dieksekusi
3. **Reload** systemd untuk membaca perubahan
4. **Start & enable** timer

### 🇬🇧 English
**Systemd** can be used to run processes or scripts at a specific time or at regular intervals. Unlike cron, systemd uses **two separate files**: a **timer** (when to run) and a **service** (what to run).

Steps to create a scheduled task with systemd:
1. Create a **timer** file (`.timer`) — defines when the service runs
2. Create a **service** file (`.service`) — defines what gets executed
3. **Reload** systemd to read the changes
4. **Start & enable** the timer

### 🇯🇵 日本語
**Systemd**は特定の時刻や定期的な間隔でプロセスやスクリプトを実行するために使えます。cronと異なり、systemdは**2つの別ファイル**を使います：**タイマー**（いつ実行するか）と**サービス**（何を実行するか）。

systemdでスケジュールタスクを作成する手順：
1. **タイマー**ファイル（`.timer`）を作成 — サービスをいつ実行するかを定義
2. **サービス**ファイル（`.service`）を作成 — 何を実行するかを定義
3. systemdを**リロード**して変更を読み込む
4. タイマーを**起動・有効化**する

---

### Step 1 — Create the Timer

```bash
# Create directory and timer file
$ sudo mkdir /etc/systemd/system/mytimer.timer.d
$ sudo vim /etc/systemd/system/mytimer.timer
```

```ini
# /etc/systemd/system/mytimer.timer

[Unit]
Description=My Timer

[Timer]
OnBootSec=3min        # Run 3 minutes after system boot
OnUnitActiveSec=1hour # Then repeat every 1 hour

[Install]
WantedBy=timers.target
```

| Option | 🇮🇩 Keterangan | 🇬🇧 Description | 🇯🇵 説明 |
|--------|--------------|----------------|---------|
| `OnBootSec=` | Jalankan N menit/detik setelah boot | Run N min/sec after boot | 起動後N分/秒後に実行 |
| `OnUnitActiveSec=` | Ulangi setiap N interval | Repeat every N interval | N間隔ごとに繰り返し |
| `WantedBy=timers.target` | Daftarkan ke target timer | Register with timers target | タイマーターゲットに登録 |

> 💡 Use `OnBootSec` alone if you want the script to run **once** after boot. Add `OnUnitActiveSec` for **recurring** execution.

---

### Step 2 — Create the Service

```bash
# Create the service file
$ sudo vim /etc/systemd/system/mytimer.service
```

```ini
# /etc/systemd/system/mytimer.service

[Unit]
Description=My Service

[Service]
ExecStart=/full/path/to/my/script.sh

[Install]
WantedBy=multi-user.target
```

| Option | 🇮🇩 Keterangan | 🇬🇧 Description | 🇯🇵 説明 |
|--------|--------------|----------------|---------|
| `Description=` | Deskripsi service | Service description | サービスの説明 |
| `ExecStart=` | Path lengkap ke script | Full path to the script | スクリプトへのフルパス |
| `WantedBy=multi-user.target` | Aktif di mode multi-user normal | Active in normal multi-user mode | 通常のマルチユーザーモードで有効 |

---

### Step 3 — Reload, Start & Enable

```bash
# Reload systemd to detect new files
$ sudo systemctl daemon-reload

# Start the timer immediately
$ sudo systemctl start mytimer.timer

# Enable timer to persist across reboots
$ sudo systemctl enable mytimer.timer

# Verify the timer is active
$ systemctl status mytimer.timer

# List all active timers
$ systemctl list-timers
```

## 3. 🕐 Cron — Schedule Tasks with Crontab

### 🇮🇩 Bahasa Indonesia
**Cron** adalah tool lain untuk menjadwalkan dan mengotomatisasi proses di Linux. Cron menggunakan file bernama **crontab** yang berisi daftar tugas dan jadwal kapan dieksekusi. Cron daemon (`crond`) membaca file ini dan menjalankan perintah sesuai jadwal.

Perbedaan utama dari systemd: Cron lebih **sederhana dan cepat dikonfigurasi** — cukup satu file untuk mendefinisikan jadwal dan perintah.

### 🇬🇧 English
**Cron** is another tool for scheduling and automating processes in Linux. Cron uses a file called **crontab** that contains a list of tasks and when they should be executed. The cron daemon (`crond`) reads this file and runs commands according to the schedule.

Main difference from systemd: Cron is **simpler and faster to configure** — just one file to define both the schedule and the command.

### 🇯🇵 日本語
**Cron**はLinuxでプロセスをスケジュールおよび自動化するもう一つのツールです。Cronは**crontab**というファイルを使い、タスクのリストといつ実行するかを記述します。cronデーモン（`crond`）がこのファイルを読み込み、スケジュールに従ってコマンドを実行します。

systemdとの主な違い：Cronは**よりシンプルで設定が速い** — スケジュールとコマンドを1つのファイルで定義できます。

```bash
# Edit the current user's crontab
$ crontab -e

# View the current user's crontab
$ crontab -l

# Remove the current user's crontab
$ crontab -r

# Edit crontab for a specific user (as root)
$ sudo crontab -u username -e

# View system-wide crontab
$ cat /etc/crontab

# View system cron jobs (directories)
$ ls /etc/cron.d/
$ ls /etc/cron.daily/
$ ls /etc/cron.weekly/
$ ls /etc/cron.monthly/
```

## 4. 📐 Cron Syntax — Format & Fields

### 🇮🇩 Bahasa Indonesia
Setiap baris dalam crontab mengikuti **format 5 field waktu** diikuti perintah yang akan dieksekusi. Urutan field harus diikuti dengan tepat.

### 🇬🇧 English
Each line in a crontab follows a **5 time-field format** followed by the command to execute. The field order must be followed exactly.

### 🇯🇵 日本語
crontabの各行は**5つの時間フィールドのフォーマット**に従い、その後に実行するコマンドが続きます。フィールドの順序を正確に守る必要があります。

```
# ┌───── minute (0-59)
# │ ┌───── hour (0-23)
# │ │ ┌───── day of month (1-31)
# │ │ │ ┌───── month (1-12)
# │ │ │ │ ┌───── day of week (0-7) [0 and 7 = Sunday]
# │ │ │ │ │
# * * * * *  /path/to/command
```

| Field | 🇮🇩 Keterangan | 🇬🇧 Description | 🇯🇵 説明 | Range |
|-------|--------------|----------------|---------|-------|
| Minute | Menit ke berapa | Which minute | 何分 | 0–59 |
| Hour | Jam ke berapa | Which hour | 何時 | 0–23 |
| Day of Month | Tanggal berapa | Which day of the month | 何日 | 1–31 |
| Month | Bulan ke berapa | Which month | 何月 | 1–12 |
| Day of Week | Hari apa | Which day of the week | 何曜日 | 0–7 (0&7=Sun) |

### 🔣 Special Characters

| Character | 🇮🇩 Arti | 🇬🇧 Meaning | 🇯🇵 意味 | Example |
|-----------|---------|------------|---------|---------|
| `*` | Setiap nilai | Every value | すべての値 | `* * * * *` = every minute |
| `*/n` | Setiap n unit | Every n units | n単位ごと | `*/6` = every 6 hours |
| `n` | Tepat nilai n | Exactly value n | 正確にn | `0` = at minute 0 |
| `,` | Beberapa nilai | Multiple values | 複数の値 | `1,15` = 1st and 15th |
| `-` | Rentang nilai | Range of values | 値の範囲 | `1-5` = Mon through Fri |

## 5. 📋 Cron Examples — Common Schedules

### 🇮🇩 Bahasa Indonesia
Berikut contoh crontab umum yang digunakan untuk berbagai tugas otomatis:

### 🇬🇧 English
Here are common crontab examples used for various automated tasks:

### 🇯🇵 日本語
様々な自動化タスクで使われる一般的なcrontabの例：

```bash
# ── EXAMPLE CRONTAB ─────────────────────────────────────────

# System Update — run every 6 hours (at minute 0)
0 */6 * * *   /path/to/update_software.sh

# Execute Scripts — run every 1st of the month at midnight
0 0 1 * *     /path/to/scripts/run_scripts.sh

# Cleanup DB — run every Sunday at midnight (0 = Sunday)
0 0 * * 0     /path/to/scripts/clean_database.sh

# Backups — run every Sunday at midnight (7 = Sunday)
0 0 * * 7     /path/to/scripts/backup.sh
```

### 📅 Schedule Quick Reference

| Cron Expression | 🇮🇩 Jadwal | 🇬🇧 Schedule | 🇯🇵 スケジュール |
|----------------|-----------|-------------|----------------|
| `* * * * *` | Setiap menit | Every minute | 毎分 |
| `0 * * * *` | Setiap jam (menit 0) | Every hour (at minute 0) | 毎時（0分） |
| `0 0 * * *` | Setiap tengah malam | Every midnight | 毎深夜0時 |
| `0 */6 * * *` | Setiap 6 jam | Every 6 hours | 6時間ごと |
| `0 0 1 * *` | Setiap tanggal 1 | Every 1st of the month | 毎月1日 |
| `0 0 * * 0` | Setiap Minggu tengah malam | Every Sunday midnight | 毎週日曜深夜 |
| `0 9 * * 1-5` | Setiap hari kerja jam 9 pagi | Every weekday at 9am | 平日毎朝9時 |
| `*/15 * * * *` | Setiap 15 menit | Every 15 minutes | 15分ごと |
| `0 0 * * 1` | Setiap Senin tengah malam | Every Monday midnight | 毎週月曜深夜 |

## 6. ⚖️ Systemd vs. Cron

### 🇮🇩 Bahasa Indonesia
Baik **systemd** maupun **cron** bisa digunakan untuk menjadwalkan dan mengotomatisasi proses di Linux. Perbedaan utamanya terletak pada **cara konfigurasi** dan tingkat kontrol yang ditawarkan.

### 🇬🇧 English
Both **systemd** and **cron** can be used to schedule and automate processes in Linux. The key difference lies in **how they are configured** and the level of control they offer.

### 🇯🇵 日本語
**systemd** と **cron** はどちらもLinuxでプロセスをスケジュールおよび自動化するために使えます。主な違いは**設定方法**と提供される制御レベルにあります。

| Feature | Systemd Timer | Cron |
|---------|--------------|------|
| 🇮🇩 File konfigurasi | 2 file (.timer + .service) | 1 file (crontab) |
| 🇬🇧 Config files | 2 files (.timer + .service) | 1 file (crontab) |
| 🇯🇵 設定ファイル | 2ファイル（.timer + .service） | 1ファイル（crontab） |
| 🇮🇩 Kompleksitas | Lebih kompleks | Lebih sederhana |
| 🇬🇧 Complexity | More complex | Simpler |
| 🇯🇵 複雑さ | より複雑 | よりシンプル |
| 🇮🇩 Logging | Terintegrasi dengan journalctl | Perlu konfigurasi manual |
| 🇬🇧 Logging | Integrated with journalctl | Needs manual setup |
| 🇯🇵 ログ | journalctlと統合 | 手動設定が必要 |
| 🇮🇩 Dependency | Bisa bergantung pada service lain | Tidak bisa |
| 🇬🇧 Dependencies | Can depend on other services | Cannot |
| 🇯🇵 依存関係 | 他のサービスに依存可能 | 不可 |
| 🇮🇩 Kasus penggunaan ideal | Tasks kompleks, service-aware | Tasks sederhana & cepat dikonfigurasi |
| 🇬🇧 Ideal use case | Complex tasks, service-aware | Simple tasks & quick setup |
| 🇯🇵 理想的なユースケース | 複雑なタスク、サービス連携 | シンプルなタスク・素早い設定 |

## 7. 🔐 Security Note — Task Scheduling as an Attack Vector

### 🇮🇩 Bahasa Indonesia
Dalam konteks **penetration testing** dan **incident response**, task scheduling adalah salah satu lokasi pertama yang diperiksa saat mencari **persistensi** atau **aktivitas berbahaya**. Penyerang sering menyalahgunakan cron dan systemd untuk:

- Menjaga **backdoor** tetap aktif secara berkala
- Mengeksekusi **reverse shell** pada interval tertentu
- Menjalankan **script berbahaya** setelah reboot

### 🇬🇧 English
In the context of **penetration testing** and **incident response**, task scheduling is one of the first places to check when looking for **persistence** or **malicious activity**. Attackers often abuse cron and systemd to:

- Keep a **backdoor** alive at regular intervals
- Execute a **reverse shell** on a schedule
- Run **malicious scripts** after reboot

### 🇯🇵 日本語
**ペンテスト**と**インシデントレスポンス**の文脈では、タスクスケジューリングは**永続性**や**悪意のある活動**を探す際に最初に確認する場所の一つです。攻撃者はcronとsystemdを悪用して以下を行うことがあります：

- 定期的に**バックドア**を維持する
- スケジュールで**リバースシェル**を実行する
- 再起動後に**悪意のあるスクリプト**を実行する

```bash
# ── CHECK FOR SUSPICIOUS SCHEDULED TASKS ────────────────────

# Check all crontabs (current user)
$ crontab -l

# Check root's crontab
$ sudo crontab -l

# Check system-wide cron jobs
$ cat /etc/crontab
$ ls -la /etc/cron.d/
$ ls -la /etc/cron.daily/
$ ls -la /etc/cron.weekly/
$ ls -la /etc/cron.hourly/

# Check all user crontabs (as root)
$ for user in $(cut -f1 -d: /etc/passwd); do \
    echo "=== $user ==="; crontab -u $user -l 2>/dev/null; \
  done

# Check suspicious systemd timers
$ systemctl list-timers --all

# Check for unusual systemd service files
$ ls -la /etc/systemd/system/
$ ls -la /home/*/.config/systemd/user/
```

> ⚠️ **Pentesting note:** Always audit `/etc/cron*`, `/var/spool/cron/`, and `systemctl list-timers` during post-exploitation to find attacker persistence mechanisms — and during red team exercises to validate your own implants.

## 🔗 Practical Workflow Example

```bash
# ── SCENARIO: Set up automated backup with both methods ──────

# ╔══ METHOD 1: CRON ═══════════════════════════════════════╗

# 1. Open crontab editor
$ crontab -e

# 2. Add a daily backup job at 2 AM
0 2 * * * /home/user/scripts/backup.sh >> /var/log/backup.log 2>&1

# 3. Verify the cron job was saved
$ crontab -l

# 4. Check system cron logs
$ grep CRON /var/log/syslog | tail -20


# ╔══ METHOD 2: SYSTEMD TIMER ══════════════════════════════╗

# 1. Create the backup script
$ sudo vim /usr/local/bin/backup.sh
$ sudo chmod +x /usr/local/bin/backup.sh

# 2. Create the systemd service
$ sudo vim /etc/systemd/system/backup.service

# --- backup.service content ---
# [Unit]
# Description=Daily Backup Service
#
# [Service]
# ExecStart=/usr/local/bin/backup.sh
#
# [Install]
# WantedBy=multi-user.target

# 3. Create the systemd timer
$ sudo vim /etc/systemd/system/backup.timer

# --- backup.timer content ---
# [Unit]
# Description=Daily Backup Timer
#
# [Timer]
# OnCalendar=*-*-* 02:00:00
# Persistent=true
#
# [Install]
# WantedBy=timers.target

# 4. Reload systemd, start and enable timer
$ sudo systemctl daemon-reload
$ sudo systemctl start backup.timer
$ sudo systemctl enable backup.timer

# 5. Verify timer is scheduled
$ systemctl list-timers | grep backup

# 6. Check logs via journalctl
$ journalctl -u backup.service --no-pager
```

## 🧠 Quick Reference Cheatsheet

```bash
# ── CRON ─────────────────────────────────────────────────────
crontab -e                               # edit current user's crontab
crontab -l                               # list current crontab
crontab -r                               # remove current crontab
sudo crontab -u <user> -e               # edit specific user's crontab
cat /etc/crontab                         # view system-wide crontab
ls /etc/cron.d/                          # view system cron jobs

# ── CRON SYNTAX ──────────────────────────────────────────────
# MIN HOUR DOM MON DOW  COMMAND
  0   */6  *   *   *    /path/script.sh  # every 6 hours
  0    0   1   *   *    /path/script.sh  # every 1st of month
  0    0   *   *   0    /path/script.sh  # every Sunday midnight
  *    *   *   *   *    /path/script.sh  # every minute

# ── SYSTEMD TIMER ────────────────────────────────────────────
sudo vim /etc/systemd/system/<name>.timer    # create timer file
sudo vim /etc/systemd/system/<name>.service  # create service file
sudo systemctl daemon-reload                 # reload after changes
sudo systemctl start <name>.timer            # start timer
sudo systemctl enable <name>.timer           # enable on boot
systemctl list-timers                        # list all timers
systemctl status <name>.timer                # check timer status
journalctl -u <name>.service                 # view service logs

# ── SECURITY AUDIT ───────────────────────────────────────────
crontab -l                               # check user crontab
sudo crontab -l                          # check root crontab
cat /etc/crontab                         # system crontab
ls /etc/cron.*                           # all system cron dirs
systemctl list-timers --all              # all systemd timers
ls /etc/systemd/system/*.timer           # list timer files
```
